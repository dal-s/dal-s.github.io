Review之每分钟处理1亿个请求

发表于 2019-01-23 | 分类于 ARTS | | 阅读次数:

原文链接在这里，不知道需不需要翻墙。

文章作者是一个网站的老大，拥有复杂的技术背景，能数的出来的编程语言都有涉猎，经过多年的浸淫，他觉得应该用合适的工具做简单的事情解决复杂的问题。

他写文章的时候，已经用了2年多的golang了。

很幸运了，接下来他就遇到了一些复杂的问题。

他们写了一个网站后端程序，用来接收网络请求，然后调用UploadToS3将请求上传的S3。

呐，我们初学者都知道嘛，go在处理并发请求是有天然的优势的，一个go func 完美解决所有问题。对的，他也是这么干的。

很简单，对不对。嗯，他的注释中也写了，不要这么干，因为请求数量太多，代码中没有控制并发数量，很快他们的程序就崩溃掉了。嗯，简直和我第一次用这个方法的时候一模一样，信心满满的运行，突如其来的崩溃。

当然了，作为一名成熟的工程师，是不可能被问题打败的嘛，作者继续优化代码，很快就上了version 2。

这次就很厉害了，这次作者用上了带缓冲的channel，呐，作为初学者，我们也知道，他要控制并发数量了嘛，防止巨量的请求把系统搞垮。

你看，他写了一个队列，然后结合select读出job，然后调用UploadToS3。嗯，作者注释中也写了，还是不够好，不过应该能够工作了，至少不会被巨量请求搞垮系统了。

正如你所想，这个代码确实不会一下子崩溃掉了，但是由于巨量的数据请求一下子涌入，很快他们的队列就满了，而且UploadToS3这个方法的出力速度远远小于请求的增长速度，他们的队列始终处于爆满的状态。然后后面的请求就阻塞掉了。虽然系统没有崩溃，但是前段请求都被阻塞了。

再接再厉，我们继续看看作者大大如何在这个版本的基础上继续优化。

他们想到了用两个channel，一个控制请求队列，一个控制UploadToS3工作。

以下是个人理解了，英文水平有限，而且代码没有完全理解。

先从初始化来看，先调用worker.Start()创建了一定数量的工作池，然后启动dispatch() 从JobQueue读出需要处理的内容，将待处理的job放入jobChannel ，由第一步创建好的工作池读出job，调用UploadToS3处理。payloadHandler 这个函数其实算是业务代码，接收请求，然后把请求放入JobQueue。

这个设计将接收请求和处理请求分开，并都使用channel并发处理，而且在系统初始化期间，创建好一定数量的线程池，等待处理。
